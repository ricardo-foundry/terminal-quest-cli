# terminal-quest-cli v2.5 — release notes

> v2.5 is the **release-readiness** milestone. There are no new game
> systems; instead it closes every TODO from v2.4, hardens the cloud
> backend, ships a real animated demo and prepares the repo for npm
> publish + GitHub social card.

## Highlights

- **Cloud sync is now update-in-place.** `--cloud push <slot>` now
  PATCHes the same gist instead of creating a fresh one each time.
  The slot → gist-id mapping lives in `~/.terminal-quest/cloud-meta.json`.
- **Community quests hot-reload in `--dev` mode.** Editing
  `quests/<id>/quest.json` while the game is running now reloads the
  pack (best-effort `fs.watch`, coalesced 150ms).
- **`minAlignment` survives a restart.** Previously the field reset to
  the *current* alignment on every fresh process — meaning the
  pacifist achievement could be falsely awarded after a single
  forgiving session. Now persisted via a `minAlignmentInit` flag in
  the save envelope.
- **`--new` keeps a `.bak`.** Before archiving `<slot>.json` to a
  timestamped sibling, we also copy it to `<slot>.json.bak`, so an
  `oops --new` is recoverable without needing the timestamp.
- **Animated SVG demo.** `docs/terminal-demo.svg` is a 12-second,
  4-scene, pure-SVG terminal animation referenced from the README.
  Generated by `scripts/record-session.js` (no GIF tooling needed).

## Known follow-ups (not blockers)

- The fs.watch hot-reload uses `recursive: true`, which is unsupported on
  some Linux kernels. It silently no-ops there — callers can still type
  `:reload-quests` (TODO for v2.6) to trigger a manual rescan.
- Cloud meta is per-host; rotating laptops still re-uploads. A future
  `--cloud sync <slot>` would compare `updatedAt` and only touch the
  slower side.
- We have no end-to-end test that exercises the full bin → game →
  save → reload pipeline. Issue tracker entry filed.
